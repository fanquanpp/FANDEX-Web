## 1. 能力维度

### 1.1 四维能力模型

```
         技术深度
            ▲
            │
            │
影响力 ◄────┼────► 技术广度
            │
            │
            ▼
         业务理解
```

### 1.2 各维度说明

| 维度     | 说明               | 度量方式               |
| :------- | :----------------- | :--------------------- |
| 技术深度 | 核心领域的专业程度 | 解决复杂问题、技术决策 |
| 技术广度 | 跨领域的技术视野   | 跨团队协作、技术选型   |
| 业务理解 | 对业务的理解深度   | 业务方案设计、业务创新 |
| 影响力   | 对组织和行业的影响 | 知识传播、标准制定     |

## 2. 各级别能力要求

### 2.1 能力矩阵

| 级别 | 技术深度   | 技术广度 | 业务理解 | 影响力 |
| :--- | :--------- | :------- | :------- | :----- |
| P5   | 单技术栈   | —        | 理解需求 | —      |
| P6   | 多技术栈   | 相邻领域 | 理解业务 | 团队内 |
| P7   | 领域专家   | 跨领域   | 驱动业务 | 跨团队 |
| P8   | 多领域深度 | 系统级   | 业务创新 | 跨部门 |
| P9   | 行业级     | 战略级   | 战略影响 | 公司级 |

### 2.2 关键跃迁

| 跃迁  | 核心变化             |
| :---- | :------------------- |
| P5→P6 | 从需要指导到独立负责 |
| P6→P7 | 从执行者到决策者     |
| P7→P8 | 从模块影响到系统影响 |
| P8→P9 | 从公司影响到行业影响 |

## 3. 影响力建设

### 3.1 影响力层次

| 层次   | 说明         | 方式                 |
| :----- | :----------- | :------------------- |
| 团队内 | 团队成员认可 | 技术分享、代码审查   |
| 跨团队 | 其他团队认可 | 技术方案、跨团队协作 |
| 跨部门 | 部门级认可   | 技术标准、最佳实践   |
| 公司级 | 公司级认可   | 技术委员会、开源项目 |
| 行业级 | 行业认可     | 技术演讲、开源贡献   |

### 3.2 影响力建设方法

| 方法     | 说明         |
| :------- | :----------- |
| 技术分享 | 定期内部分享 |
| 技术博客 | 公开技术文章 |
| 开源贡献 | 贡献开源项目 |
| 技术演讲 | 行业会议演讲 |
| 标准制定 | 制定技术规范 |

## 4. 晋升准备策略

### 4.1 晋升答辩

| 准备     | 说明           |
| :------- | :------------- |
| 业绩梳理 | 量化核心贡献   |
| 能力证明 | 展现下一级能力 |
| 未来规划 | 清晰的发展方向 |
| 答辩练习 | 模拟答辩场景   |

### 4.2 常见误区

| 误区       | 正确做法         |
| :--------- | :--------------- |
| 做了就够   | 还要证明做了什么 |
| 苦劳=功劳  | 关注结果和影响   |
| 只关注技术 | 业务理解同样重要 |
| 等待被发现 | 主动展示和沟通   |
