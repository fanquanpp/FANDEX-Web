### 1. 关于进程和线程，以下说法正确的是？

A. 线程是资源分配的基本单位
B. 进程是 CPU 调度的基本单位
C. 同一进程的线程共享堆内存
D. 线程切换的开销大于进程切换

<details>
<summary>查看答案</summary>

**答案**: C

**解析**: 进程是资源分配的基本单位，线程是 CPU 调度的基本单位（A、B 说反了）。同一进程的线程共享代码段、数据段、堆内存，但各自有独立的栈和寄存器。线程切换只需保存/恢复寄存器和栈指针，开销远小于进程切换（需切换地址空间等）。
</details>

### 2. TCP 三次握手的正确顺序是？

A. SYN → SYN+ACK → ACK
B. SYN → ACK → SYN+ACK
C. SYN+ACK → SYN → ACK
D. ACK → SYN → SYN+ACK

<details>
<summary>查看答案</summary>

**答案**: A

**解析**: 三次握手流程：① 客户端发送 SYN（seq=x）；② 服务器回复 SYN+ACK（seq=y, ack=x+1）；③ 客户端发送 ACK（ack=y+1）。三次握手确保双方都确认了对方的发送和接收能力。
</details>

### 3. 关于 Cache，以下说法错误的是？

A. Cache 利用的是程序的局部性原理
B. Cache 命中率通常用 LRU 算法来提升
C. Cache 和主存的地址映射由硬件完成
D. Cache 容量越大，命中率一定越高

<details>
<summary>查看答案</summary>

**答案**: D

**解析**: Cache 容量增大通常能提高命中率，但并非"一定"——当容量增大到一定程度后，命中率提升趋于平缓（边际递减），且更大的 Cache 可能导致访问延迟增加。此外，工作集超出 Cache 容量时命中率仍会骤降。
</details>

### 5. 在 HTTP/1.1 中，以下哪个头部字段用于实现持久连接？

A. `Content-Length`
B. `Connection: keep-alive`
C. `Transfer-Encoding: chunked`
D. `Cache-Control`

<details>
<summary>查看答案</summary>

**答案**: B

**解析**: `Connection: keep-alive` 表示 TCP 连接在请求完成后不关闭，可复用于后续请求，避免反复建立连接的开销。HTTP/1.1 默认持久连接。`Content-Length` 标识消息体长度，`Transfer-Encoding: chunked` 用于分块传输，`Cache-Control` 控制缓存策略。
</details>

## 简答题

### 1. 解释虚拟内存的工作原理

请说明虚拟内存的概念、页表的作用、TLB 的加速机制，以及缺页中断的处理流程。

<details>
<summary>查看参考答案</summary>

**虚拟内存**：为每个进程提供独立的虚拟地址空间，使程序可使用超过物理内存的地址范围。虚拟地址通过页表映射到物理地址。

**页表**：存储虚拟页号到物理页框号的映射关系，每个进程有独立的页表。页表项包含物理页框号、有效位、修改位、访问位等。

**TLB（快表）**：页表的 Cache，存储最近使用的页表项。地址转换时先查 TLB（命中则直接得到物理地址），未命中再查页表（并将结果装入 TLB）。TLB 命中率通常 > 95%，极大加速地址转换。

**缺页中断流程**：
1. CPU 访问虚拟地址，查 TLB 未命中
2. 查页表，发现有效位为 0（页不在内存中）
3. 触发缺页中断，陷入内核态
4. 操作系统在磁盘上找到所需页面
5. 选择一个物理页框（可能需要页面置换算法如 LRU）
6. 若被替换页被修改过，先写回磁盘
7. 将所需页面从磁盘读入物理页框
8. 更新页表项，设置有效位为 1
9. 返回用户态，重新执行引发缺页的指令
</details>

### 2. 对比 TCP 和 UDP 的区别，并说明各自适用的场景

<details>
<summary>查看参考答案</summary>

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接方式 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠传输（确认、重传、序号） | 不可靠（尽最大努力交付） |
| 有序性 | 保证数据有序到达 | 不保证顺序 |
| 流量控制 | 滑动窗口机制 | 无 |
| 拥塞控制 | 慢启动、拥塞避免等 | 无 |
| 传输效率 | 较低（头部 20 字节起） | 较高（头部 8 字节） |
| 通信模式 | 一对一 | 一对一、一对多、多对多 |

**TCP 适用场景**：
- Web 浏览（HTTP/HTTPS）
- 文件传输（FTP）
- 电子邮件（SMTP/IMAP）
- 需要数据完整性的场景

**UDP 适用场景**：
- 实时音视频（WebRTC、直播）
- 在线游戏（低延迟优先）
- DNS 查询
- IoT 传感器数据上报
- 广播/多播通信
</details>

### 3. 解释编译型语言和解释型语言的区别，以及 JIT 编译如何弥合两者

<details>
<summary>查看参考答案</summary>

**编译型语言**（如 C/C++/Go）：
- 源代码在运行前通过编译器一次性翻译为机器码
- 优点：运行速度快，无需运行时翻译开销
- 缺点：平台相关，编译-运行周期长，调试迭代慢

**解释型语言**（如 Python/Ruby/JavaScript）：
- 源代码在运行时由解释器逐行翻译执行
- 优点：跨平台，开发迭代快，动态特性丰富
- 缺点：运行速度慢，每次执行都需重新翻译

**JIT（Just-In-Time）编译**：
- 结合两者优点，在程序运行时将热点代码（频繁执行的代码）编译为机器码
- 工作流程：先解释执行 → 性能分析识别热点 → 将热点代码编译为机器码 → 后续直接执行机器码
- 典型实现：V8（JavaScript）、PyPy（Python）、JVM（Java）、LuaJIT
- 优势：启动快（解释执行），长期运行性能接近编译型语言
- 代价：JIT 编译本身消耗时间和内存，有"预热"阶段
</details>
